# Phase 5: Quality Hardening & MVP Release

## Objective
Achieve all five coverage gates simultaneously (QG-001 ≥90% unit, QG-002 ≥80% E2E aggregate, QG-003 ≥50% CLI E2E, QG-004 ≥50% TUI E2E, QG-005 ≥50% MCP E2E), 100% requirement traceability, clean security audit (`cargo audit --deny warnings`), zero BOOTSTRAP-STUB annotations, and empty risk matrix violations. This phase is executed primarily by AI agents using the agentic development loop established in Phase 4. All work follows mandatory TDD Red-Green-Refactor discipline. Performance benchmarks and SLO verification are completed. The phase culminates in the MVP release milestone.

## Detailed Deliverables & Components

### E2E Test Suite (All Interfaces)
- CLI E2E tests via `assert_cmd` subprocess invocation achieving ≥50% line coverage (QG-003)
- TUI E2E tests via `TestBackend` + full `handle_event→render` cycle achieving ≥50% line coverage (QG-004)
- MCP E2E tests via `POST /mcp/v1/call` HTTP requests achieving ≥50% line coverage (QG-005)
- Aggregate E2E coverage ≥80% across all interfaces (QG-002)
- Each test annotated with `// Covers: REQ-ID` for traceability
- Tests must use actual interface boundaries (ROAD-CONS-006)
- Expected behavior: All five QG gates pass simultaneously in single `./do coverage` invocation

### Unit Test Coverage Hardening
- All crates achieve ≥90% line coverage from unit tests (QG-001)
- Coverage gaps identified from `target/coverage/report.json` and targeted with specific tests
- TDD discipline: write failing test first, then implement
- Expected behavior: `cargo llvm-cov` reports ≥90% for every crate

### 100% Requirement Traceability
- Every requirement ID has at least one `// Covers:` annotation in test code
- `target/traceability.json` reports `traceability_pct == 100.0` and `stale_annotations` is empty
- `./do test` generates traceability report; exits non-zero if incomplete
- Expected behavior: `./do test` reports full traceability

### Security Audit & Hardening
- `cargo audit --deny warnings` exits 0 on all 3 platforms
- No CRITICAL or HIGH severity advisories suppressed without expiry date and justification in `audit.toml`
- Security design verification: audit events, credential handling, file permissions, subprocess isolation
- Adapter compatibility documentation in `docs/adapter-compatibility.md`
- Expected behavior: Clean security audit on all platforms

### BOOTSTRAP-STUB Resolution
- Zero `// TODO: BOOTSTRAP-STUB` annotations remain in any `.rs` file
- `./do lint` enforces this after Phase 3 PTC is committed
- Each stub resolved with targeted implementation via TDD, not annotation removal
- Expected behavior: `grep -r 'BOOTSTRAP-STUB' crates/` returns no results

### Performance Benchmarks & SLO Verification
- Performance SLOs defined and verified through automated benchmarks
- Performance regression testing framework
- Performance trace collection and analysis
- Performance budget enforcement per component
- Benchmark results tracked in CI artifacts
- Expected behavior: All performance SLOs met; no regressions from baseline

### KPI Business Rules
- All KPI thresholds validated through automated tests
- Development standards enforcement: auto-formatting, linting, doc comments, coverage
- Expected behavior: All KPI gates pass

### Risk Matrix Completion
- `target/risk_matrix_violations` array is empty
- All risks with score ≥6 have covering tests
- No missing mitigations; no uncovered high-score risks
- Per-crate ADRs committed to `docs/adr/`
- `docs/adr/fallback-registry.json` tracks all fallback states
- Expected behavior: `./do test` exits 0 with clean risk matrix

### MVP Release Gate
- All PTC-5-* conditions pass simultaneously:
  - PTC-5-001: All five QG gates pass in single `./do coverage` invocation
  - PTC-5-002: 100% traceability with no stale annotations
  - PTC-5-004: `cargo audit --deny warnings` exits 0 on all 3 platforms
  - PTC-5-005: Zero BOOTSTRAP-STUB annotations
  - PTC-5-006: Empty risk matrix violations
  - PTC-5-007: `./do presubmit` exits 0 within 900 seconds on all 3 platforms
- MVP release milestone tagged in git
- Expected behavior: Phase 5 PTC committed; release milestone created

## Technical Considerations
- All Phase 5 work is executed by AI agents following TDD discipline (Red-Green-Refactor)
- Coverage gaps are identified systematically from `target/coverage/report.json` and targeted with specific tests
- E2E tests are the most time-consuming to write and maintain — prioritize high-coverage tests first
- QG-005 (MCP E2E ≥50%) is historically the hardest gate to achieve — plan extra test effort
- Performance testing must not cause presubmit to exceed the 900-second budget
- Traceability annotations must exactly match active requirement IDs — stale annotations are invalid
- Security audit may reveal dependency vulnerabilities requiring version upgrades — budget time for this
