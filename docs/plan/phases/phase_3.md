# Phase 3: Server & Client Interfaces

## Objective
Build the server binary (`devs-server`) integrating the gRPC API (`devs-grpc`) and Glass-Box MCP server (`devs-mcp`), then implement all three client interfaces: CLI (`devs-cli`), TUI (`devs-tui`), and MCP stdio bridge (`devs-mcp-bridge`). The server binds gRPC on port 7890 and MCP on port 7891, writes the discovery file, and handles graceful shutdown. The MCP server exposes all 20 Glass-Box tools for agentic development. The TUI provides a tabbed dashboard with split panes. The CLI implements all MVP commands. The MCP bridge forwards JSON-RPC between stdio and the MCP server. All Bootstrap Phase COND-001 conditions are met. This is the largest phase, encompassing all user-facing interface design, UI/UX architecture, visual design, and accessibility requirements.

### devs-server Crate (Startup/Shutdown)
- Server startup sequence: bind gRPC port → validate config → bind MCP port → initialize pools → restore checkpoints → write discovery file
- Graceful shutdown: cancel in-flight stages → checkpoint running state → close gRPC streams → delete discovery file
- Discovery file: atomic write to well-known path (~/.config/devs/server.addr); DEVS_DISCOVERY_FILE override for testing
- Server configuration via devs.toml with CLI flag and environment variable overrides
- Health check endpoint for client liveness verification
- Expected behavior: Server starts, binds both ports, writes discovery file on all 3 platforms; clean shutdown on SIGTERM

### devs-cli Crate (Command-Line Client)

- Output formats: human-readable (default) and JSON (--format json)
- Server discovery: explicit --server flag or auto-discovery via discovery file
- Exit codes: 0 = success, 1 = general error, 2 = usage error, 4 = validation error
- Streaming log output for `devs logs --follow`
- Cross-platform path normalization (Windows backslash handling)
- Expected behavior: All CLI commands work correctly against running server

## Technical Considerations
- `devs-grpc` and `devs-mcp` can be developed in parallel (both depend on `devs-scheduler`, not each other)
- `devs-cli`, `devs-tui`, and `devs-mcp-bridge` can be developed in parallel once `devs-server` is complete
- The MCP server must be active from the first commit where devs-server binds ports (ROAD-CONS-005)
- TUI testing uses TestBackend + full handle_event→render cycle, not pixel snapshots
- CLI E2E tests use assert_cmd subprocess invocation
- MCP E2E tests use POST /mcp/v1/call HTTP requests
- Cross-platform path handling is critical for Windows compatibility (ROAD-CONS-006)
- gRPC streaming RPCs must respect Tokio context cancellation
- The Glass-Box MCP server is the primary interface for agentic development — feature completeness is critical
