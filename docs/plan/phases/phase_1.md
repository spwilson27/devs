# Phase 1: Core Domain & Infrastructure

## Objective
Build the six foundational library crates that form the infrastructure layer of `devs`: `devs-config` (configuration and project registry parsing), `devs-checkpoint` (git-backed state persistence), `devs-adapters` (five agent CLI adapters with PTY support), `devs-pool` (agent pool routing, concurrency semaphore, and fallback), and `devs-executor` (execution environment management for tempdir, Docker, and remote SSH). Critical risks RISK-002 (PTY on Windows) and RISK-004 (adapter CLI breakage) are mitigated before adapter development begins. All crates achieve ≥90% unit test line coverage. Security foundations including file permissions, credential handling (`Redacted<T>`), and input validation are established. The full risk matrix with mitigations is implemented and tested.

## Detailed Deliverables & Components

### devs-config Crate
- Parse `devs.toml` server configuration: listen address, MCP port, default pool, scheduling policy, webhook targets, credential entries
- Parse pool definitions with agent configurations, capability tags, fallback flags, and concurrency limits
- Project registry management via `devs project add` CLI command with repo path, priority/weight, checkpoint branch, workflow search paths
- Configuration override chain: TOML file → environment variables → CLI flags
- Input validation collecting all errors in a single pass
- Expected behavior: All config types parse and validate correctly; 90% unit coverage

### devs-checkpoint Crate
- Git-backed checkpoint and log persistence using `git2` on `spawn_blocking`
- Configurable checkpoint branch: working branch or dedicated state branch (e.g., `devs/state`)
- Workflow definition snapshotting: immutable copy stored with checkpoint at run start
- Log retention policy enforcement (max age, max size)
- Atomic checkpoint commits with rollback on failure
- Expected behavior: Checkpoints survive server crash; 90% unit coverage

### devs-adapters Crate
- Five agent CLI adapters: Claude Code, Gemini CLI, OpenCode, Qwen, GitHub Copilot CLI
- Adapter trait (`AgentAdapter`) with flag-based and file-based prompt passing
- PTY mode support with `portable_pty` and `PTY_AVAILABLE` static flag for Windows compatibility
- Rate-limit detection: passive (exit code/stderr pattern matching) and active (MCP tool call)
- CLI flag constants in `config.rs` per adapter; `target/adapter-versions.json` provisioning via `./do setup`
- Bidirectional mid-run interaction: agent→devs via MCP, devs→agent via stdin/MCP push
- Expected behavior: All 5 adapters spawn correctly; PTY probe passes on all platforms; 90% unit coverage

### devs-pool Crate
- Named agent pool with ordered agent configurations
- Routing: priority fallback order + capability tag filtering
- Concurrency management via `tokio::sync::Semaphore` with configurable `max_concurrent`
- Rate-limit cooldown tracking with absolute `DateTime<Utc>` timestamps
- PoolExhausted event with once-per-episode webhook firing
- Expected behavior: Pool dispatch selects correct agent per capability and availability; 90% unit coverage

### devs-executor Crate
- Three execution environment targets: tempdir, Docker (`DOCKER_HOST`), remote SSH (`ssh_config`)
- Repository cloning into isolated working directory before stage runs
- Context file (`.devs_context.json`) writing to working directory
- Environment variable injection (per-stage + inherited server vars + `DEVS_MCP_ADDR`)
- Artifact collection: agent-driven (commit/push) or auto-collect (diff, commit, push)
- Working directory cleanup after stage completion
- Expected behavior: All three execution targets operational; 90% unit coverage

### Security Foundations
- File permission enforcement for config files, credentials, checkpoint data
- `Redacted<T>` wrapper type for sensitive values (API keys, credentials)
- Input validation and sanitization at system boundaries
- Credential handling: environment variables (preferred) or TOML config entries with security caveat
- Subprocess environment isolation and cleanup
- Expected behavior: No credential leaks in logs or error messages; security tests pass

### Risk Matrix & Mitigations
- Complete risk matrix implementation with severity scoring
- All risks with score ≥ 6 have covering tests (`// Covers: RISK-NNN`)
- RISK-002 mitigation: PTY platform probe test on all 3 platforms
- RISK-004 mitigation: Adapter CLI flag verification and version provisioning
- All remaining risk mitigations documented and tested
- Expected behavior: `target/risk_matrix_violations` empty; all covering tests pass

## Technical Considerations
- `devs-config`, `devs-checkpoint`, and `devs-adapters` can be developed in parallel (all depend only on `devs-core`)
- `devs-pool` requires `devs-adapters` completion; `devs-executor` requires `devs-pool` — sequential sub-chain
- `git2` operations must use `spawn_blocking` to avoid blocking the Tokio runtime
- PTY availability varies by platform; `PTY_AVAILABLE` flag enables graceful degradation on Windows
- Rate-limit cooldown stored as absolute timestamps survives server restarts
- Docker and SSH execution environments require careful cleanup on stage failure/timeout
- Security foundations (Redacted<T>, file permissions) must be in place before any credential-handling code
