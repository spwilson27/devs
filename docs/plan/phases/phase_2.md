# Phase 2: Workflow Engine

## Objective
Build the DAG-based workflow scheduling engine (`devs-scheduler`) and outbound webhook notification system (`devs-webhook`). The scheduler implements dependency-driven stage dispatch, fan-out parallelism with configurable merge handlers, per-stage retry with backoff, per-stage and workflow-level timeout enforcement, and multi-project scheduling with configurable priority policies. Stage completion signal processing (exit code, structured output, MCP tool call) is fully implemented. The webhook dispatcher provides at-least-once delivery with fixed-backoff retry. DAG dispatch latency is verified at ≤100ms. Cycle detection via Kahn's algorithm is implemented and tested.

## Detailed Deliverables & Components

### devs-scheduler Crate (DAG Engine)
- Directed acyclic graph (DAG) stage scheduling: stages declare `depends_on` lists; eligible stages dispatched automatically when all dependencies complete
- Cycle detection using Kahn's algorithm returning `{ "error": "cycle detected", "cycle": ["A","B","A"] }`
- DAG dispatch latency ≤100ms verified by monotonic clock test
- Workflow run state machine: Pending → Running → Completed/Failed
- Stage run state machine: Waiting → Eligible → Running → Completed/Failed/TimedOut/Cancelled
- Fan-out: parallel agent execution within a single stage, configurable via closure (builder API) or `fan_out` count/input list (declarative)
- Fan-out merge: default array join or custom merge handler (Rust closure / named handler)
- Per-stage retry: max retry count, backoff strategy, automatic retry before error branch
- Branch loopback retry: workflow graph error branch routes back to failed stage
- Per-stage timeout: cancel signal on expiry, stage marked as failed
- Workflow-level timeout: global cap on total run duration
- Multi-project scheduling: strict priority queue or weighted fair queuing, configurable per-project
- Workflow input parameter validation: typed inputs (string, path, integer, boolean), template variable interpolation
- Run identification: user-provided name + UUID/slug deduplication; auto-generated slug if none provided
- Workflow snapshotting: definition snapshot stored with checkpoint at run start
- Expected behavior: DAG scheduling dispatches eligible stages within 100ms; cycle detection catches circular dependencies; fan-out and merge work correctly; 90% unit coverage

### Stage Completion Signal Processing
- Three configurable mechanisms per stage: `exit_code`, `structured_output`, `mcp_tool_call`
- `exit_code`: zero = success, non-zero = failure; rate-limit detection via exit code/stderr patterns
- `structured_output`: `.devs_output.json` file or stdout last-line JSON; strict `"success"` boolean field; file takes priority over stdout
- `mcp_tool_call`: agent calls `signal_completion` MCP tool; fallback to exit_code if no signal before exit
- `signal_completion` on terminal state returns error `"failed_precondition: stage is already in a terminal state"`
- Rate-limit detection triggers 60s cooldown, stage re-queued, attempt count NOT incremented
- `StageRun.exit_code` always recorded regardless of completion mechanism
- Expected behavior: All three completion mechanisms correctly determine stage outcomes

### Data Flow Between Stages
- Template variables: `{{stage.<name>.<field>}}` syntax for referencing prior stage outputs
- Context file: full prior stage output written to `.devs_context.json` for next agent
- Shared directory convention: agents read/write to known working directory locations
- Template resolution with workflow input parameters and stage output fields
- Expected behavior: Downstream stages receive correct data from upstream stages

### Workflow Submission Validation
- 7-step atomic validation under per-project mutex: definition exists, required inputs present and type-valid, type coercion, no extra keys, run name unique, server not shutting down, create WorkflowRun
- Input type coercion for string, path, integer, boolean
- Error responses with appropriate gRPC status codes (invalid_argument, not_found, already_exists, failed_precondition)
- Expected behavior: Invalid submissions rejected with precise error messages; valid submissions create WorkflowRun atomically

### devs-webhook Crate
- Outbound webhook notifications for configurable event types: run lifecycle, stage lifecycle, pool exhaustion, all state changes
- At-least-once delivery with fixed-backoff retry
- Dedicated `tokio::sync::mpsc` channel with bounded buffer
- Notification targets and subscribed event types configured per-project
- Expected behavior: Webhooks delivered reliably; wiremock integration test validates at-least-once delivery; 90% unit coverage

### Multi-Project Scheduling
- Single server manages multiple projects simultaneously with shared agent pool
- Configurable scheduling policy: strict priority queue or weighted fair queuing
- Per-project priority/weight assignment via project registry
- Expected behavior: Higher-priority projects get first access to agents under strict priority; proportional allocation under weighted fair queuing

## Technical Considerations
- The scheduler maintains all WorkflowRun and StageRun instances in a single `Arc<RwLock<HashMap>>` — lock contention must be minimized
- Lock acquisition order must be consistent across all code paths to prevent deadlocks
- Fan-out merge handlers must handle partial failures (some parallel agents succeed, others fail)
- Template variable resolution must be safe against injection (no code execution in templates)
- Webhook delivery must not block the scheduler's main dispatch loop (dedicated channel)
- DAG dispatch latency test uses monotonic clock to avoid wall-clock skew issues
- Workflow snapshots are write-once and immutable — no mutation after run starts
