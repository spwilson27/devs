# Phase 4: Self-Hosting & Agentic Development

## Objective
Achieve the bootstrap milestone: `devs` successfully runs its own `presubmit-check` workflow on all three CI platforms (Linux, macOS, Windows). This phase creates the six standard workflow TOML files, validates the three bootstrap conditions (COND-001: server binds ports, COND-002: submit succeeds, COND-003: run completes), and activates the agentic development loop. After bootstrap, AI agents can use the Glass-Box MCP interface to develop `devs` itself. Fallback mechanisms (FB-007) are defined in case the bootstrap encounters deadlock. This is the pivot point where human developer effort transitions to AI agent effort.

## Detailed Deliverables & Components

### Standard Workflow TOML Files
- Six workflow definitions in `.devs/workflows/`:
  - `tdd-red.toml`: Red phase of TDD — run a test expecting failure
  - `tdd-green.toml`: Green phase of TDD — run implementation until test passes
  - `presubmit-check.toml`: Full presubmit pipeline as a devs workflow
  - `build-only.toml`: Compile workspace only
  - `unit-test-crate.toml`: Run unit tests for a specific crate
  - `e2e-all.toml`: Run all E2E tests across all interfaces
- All workflow files syntactically valid and accepted by `devs submit` without errors
- Workflow input parameters defined and validated
- Expected behavior: All 6 workflows parse correctly and can be submitted

### Bootstrap Validation (3-Condition Gate)
- COND-001: `devs-server` binds gRPC :7890 and MCP :7891 on all 3 platforms; discovery file written atomically; health check passes
- COND-002: `devs submit presubmit-check` via `devs-cli` exits 0 and returns valid `run_id` UUID4 on all 3 platforms
- COND-003: The `presubmit-check` workflow run reaches `status: "completed"` with all stages `status: "completed"` on all 3 platforms
- Bootstrap execution sequence: start server → verify discovery file → add project → submit presubmit-check → stream logs → verify completion
- Bootstrap ADR committed to `docs/adr/<NNNN>-bootstrap-complete.md` with CI pipeline URL, git SHA, and all 9 COND platform confirmations
- Expected behavior: Bootstrap succeeds on all 3 platforms in a single CI pipeline run

### Bootstrap Failure Protocol
- Mandatory diagnostic sequence: retrieve logs → classify failure → apply targeted fix → re-run
- Failure classification: compile error, test failure, coverage gate failure, clippy denial, traceability failure, timeout, unclassified
- Maximum 3 consecutive failures on same stage before escalating to server tracing logs
- No speculative refactoring during bootstrap debugging
- Expected behavior: Bootstrap failures diagnosed and fixed systematically

### Fallback Mechanism (FB-007)
- If bootstrap exceeds 150% of planned duration, FB-007 activates
- Fallback: `./do run-workflow` serial shell script executing workflow stages without devs server
- Fallback Activation Record (FAR) committed to `docs/adr/` before fallback script authored
- `./do presubmit` emits `WARN: Active fallback: FB-007` while fallback is active
- Fallback retired only after passing bootstrap completion PTC
- Fallback state machine and protocol rules
- Expected behavior: Fallback provides development continuity if bootstrap is blocked

### Agentic Development Loop Activation
- TDD development loop: Red (write failing test) → Green (implement until passing) → Refactor → submit presubmit-check
- Diagnostic read-before-write rule: agents must call `get_stage_output` before editing any file
- `list_runs` check before second `submit_run` for same workflow
- AI agents use Glass-Box MCP tools for development, testing, and debugging
- Expected behavior: Agents can autonomously develop, test, and iterate on devs

### Roadmap TDD and Agentic Requirements
- TDD loop enforcement via workflow definitions (tdd-red, tdd-green)
- Failure classification lookup table for agent diagnostic responses
- Bootstrap timing constraint: within 900-second presubmit budget
- E2E test suite preparation may begin in parallel with bootstrap validation
- Expected behavior: Development methodology fully automated and enforced

## Technical Considerations
- Bootstrap must NOT be attempted until `./do presubmit` passes on Linux with Phase 0 stub workspace
- The bootstrap is a final integration verification, not a debugging session for earlier phase issues
- Windows path normalization issues are the most likely bootstrap failure mode
- Each platform's bootstrap must be verified by actual CI pipeline runs, not local testing
- The fallback mechanism (FB-007) is a safety net — bootstrap success is the primary goal
- E2E test development can safely proceed in parallel with bootstrap since it doesn't modify production code
